## 1. 背景与动机

### 1.1 现有架构的核心问题

当前 Bmail 扩展采用**侵入式 DOM 注入**架构：通过 Content Script 在目标邮件网站（Gmail、网易邮箱、QQ 邮箱、Outlook）的原生页面上，精确定位并修改其 DOM 结构，向原生的发送工具栏、阅读工具栏等位置插入 Bmail 的功能按钮，拦截原生的附件上传流程，监听原生编辑区域的变化。

**这一架构存在根本性的脆弱性**：

| 问题类型 | 具体表现 |
|---------|---------|
| **强依赖第三方 DOM 结构** | 每个网站都有大量硬编码的 CSS 选择器（如 `".T-I.T-I-KE.L3"`、`".js-component-toolbar.nui-toolbar"` 等），一旦网站改版，选择器即失效 |
| **兼容性维护成本高** | 目前已有 `content_google.ts`、`content_netease.ts`、`content_outlook.ts`、`content_qq.ts` 四个独立文件，且部分网站（如 QQ 邮箱）还需同时维护新旧两个版本的逻辑 |
| **被动应对改版** | 网站改版后，只能等用户反馈再修复，存在功能中断窗口期 |
| **iframe 嵌套复杂性** | 网易邮箱、QQ 邮箱老版本等使用 iframe 作为编辑区域，需要专门处理跨 frame 的 DOM 操作，逻辑复杂且容易出错 |

### 1.2 升级目标

构建一套**以 Bmail 自有界面为核心**的邮件撰写、加解密工作流，将对第三方网站 DOM 结构的依赖降至最低，仅保留必要的"借用发送服务"接口，以此实现架构的长期稳定性。

> **范围说明（V2）**：本次升级聚焦于撰写侧（Bmail 自有面板）和阅读侧瘦化改造，**不接入第三方邮件服务的官方 API**（Gmail API / Microsoft Graph API 等），原因是其申请审批流程复杂、合规要求高，不在当前阶段范围内。

## 2. 需求总览

### 2.1 核心设计理念

> **"Bmail 管理内容，网站提供通道"**

- **邮件内容的编写、加密、附件管理**：完全在 Bmail 自有的界面和逻辑中完成
- **邮件的实际发送**：借用所在网站的发送服务（填充数据并触发原生发送动作）
- **邮件的阅读与解密**：**保留轻量注入方案**——在原生阅读界面注入解密按钮，完成语义化选择器改造，降低维护成本（详见第 5 节决策记录）

### 2.2 功能模块划分

```
┌──────────────────────────────────────────────────────────┐
│                    Bmail 自有界面层                        │
│  ┌─────────────────┐   ┌────────────────────────────────┐ │
│  │  Bmail 撰写面板  │   │      Bmail 附件管理面板          │ │
│  │  (自有编辑区域)  │   │  (加密上传 / 加密附件列表)       │ │
│  └────────┬────────┘   └───────────────┬────────────────┘ │
│           │                            │                   │
│  ┌────────▼────────────────────────────▼──────────────┐   │
│  │              Bmail 核心能力层                        │   │
│  │  加密引擎 | 解密引擎 | 附件加解密 | 联系人查询        │   │
│  └────────────────────────┬───────────────────────────┘   │
└───────────────────────────┼──────────────────────────────┘
                            │ 最小化接口
┌───────────────────────────▼──────────────────────────────┐
│                  第三方网站适配层（瘦适配器）               │
│  填充内容到发件区 | 触发原生发送 | 识别收件人邮箱地址        │
└──────────────────────────────────────────────────────────┘
```

## 3. 详细功能需求

### 3.1 Bmail 自有撰写面板

#### 3.1.1 触发方式
- 在每个支持的邮件网站上，Bmail 注入一个**固定的、不依赖原生 DOM 位置的**悬浮入口按钮或侧边栏按钮
- 用户点击该入口后，弹出 Bmail 自有的撰写面板（可以是侧边抽屉、模态弹窗或嵌入式 Panel）
- 该面板的触发入口定位策略需尽量稳定，尽量挂载在 `document.body` 层级而非网站内部具体节点上，以减少因网站改版导致的定位失效

#### 3.1.2 撰写面板核心功能
- **收件人输入**：支持手动输入邮件地址，支持从 Bmail 联系人缓存中自动补全
- **邮件正文编辑**：提供富文本编辑能力（至少支持纯文本，可扩展富文本）
- **标题输入**：提供邮件主题/标题输入区域
- **联系人地址查询**：输入收件人邮箱地址后，自动向 Bmail 联系人服务查询对应的区块链地址（联系人服务将迁移至 Cloudflare，详见 3.5 节）
- **发送前状态校验**：登录状态检查、收件人有效性校验、正文非空校验

#### 3.1.3 发送流程
```
用户点击"加密发送"
       ↓
在 Bmail 自有面板中完成加密（调用 background 的 EncryptData）
       ↓
将加密后的密文通过"适配器接口"填充到目标网站的发件区域
       ↓
通过"适配器接口"触发目标网站的发送动作
       ↓
发送完成，关闭/清空 Bmail 撰写面板
```

**关键约束**：加密动作在 Bmail 面板内完成，填充和发送由适配器完成，两者职责分离。

### 3.2 附件管理

#### 3.2.1 加密附件上传
- 在 Bmail 撰写面板内提供独立的附件上传入口（不依赖原生附件按钮）
- 用户选择文件后，Bmail 在本地完成文件加密（复用现有 `content_attachment.ts` 逻辑）
- 加密后的文件作为普通附件，通过适配器触发原生附件上传流程上传（或直接通过 Bmail 自有方式处理）

> **注**：附件上传的具体策略（是借用原生上传通道还是 Bmail 自己处理）需要架构师给出方案。

#### 3.2.2 加密附件下载与解密
- 收到含加密附件的邮件时，Bmail 需提供解密下载能力
- 解密能力的入口展示位置（原生附件列表旁 vs Bmail 自有面板内）需要架构师评估

### 3.3 邮件阅读与解密（已决策：保留轻量注入）

**决策结论**：阅读侧保留在原生界面注入解密按钮的方案，但进行语义化改造以降低维护成本。详见第 5 节决策记录。

**阅读侧核心需求**：
- 在用户打开一封含 Bmail 密文的邮件时，在邮件正文上方注入「解密」按钮
- 用户点击后，调用 background 完成解密，将解密内容展示在原生阅读区
- 各平台选择器统一改为语义化属性（`[role]`、`[aria-label]`、`[data-testid]`）
- FAB 悬浮按钮提供「解密当前邮件」备用入口，注入失败时降级使用

### 3.4 各网站适配器（瘦适配器）

每个支持的网站保留一个轻量适配器，**只负责以下最小化接口**：

| 接口职责 | 说明 |
|---------|------|
| **读取当前登录邮箱地址** | 供发送前校验使用（复用现有 `readCurrentMailAddress` 逻辑，但需尽量稳定） |
| **填充加密内容到发件区** | 将 Bmail 加密后的密文写入原生编辑区，准备发送 |
| **触发原生发送动作** | 模拟点击原生发送按钮 |
| **触发原生附件上传**（可选） | 如果附件方案选择借用原生上传通道 |

**瘦适配器的设计原则**：
- 使用尽可能少的 CSS 选择器
- 优先使用稳定属性（如 `[data-testid]`、`[role]`、`[aria-label]` 等语义化属性）而非样式类名
- 每个适配器接口应有降级处理，不能因一个接口失效导致整个功能崩溃

### 3.5 联系人服务迁移至 Cloudflare【新增需求】

#### 3.5.1 背景
当前联系人查询服务（邮箱地址 → 区块链地址的映射查询）部署在自有服务器上，通过 `background.ts` 中的 `BMRequestToSrv` 直接调用。项目方已拥有 Cloudflare 账号，需将该服务迁移至 Cloudflare 基础设施，以提升服务的可用性、访问速度和维护便利性。

#### 3.5.2 涉及的服务接口
以下现有 API 接口需迁移或通过 Cloudflare 代理：

```
API_Query_By_EMails      /query_by_email_array    // 批量邮箱地址查询区块链地址（核心）
API_Query_Bmail_Details  /query_account           // 查询账户详情
API_Bind_Email           /bind_account            // 绑定邮箱
API_Unbind_Email         /unbind_account          // 解绑邮箱
API_Active_Account       /account_active          // 激活账户
API_Active_By_Email      /active_by_email         // 通过邮箱激活
API_Decrypt_By_Admin     /decrypt_by_admin        // 管理员解密
API_Uninstall_User       /uninstall_user          // 卸载时清理
```

#### 3.5.3 架构师需要决策的内容
- **迁移方案选型**：是将服务逻辑直接部署为 **Cloudflare Workers**（无服务器函数），还是以 **Cloudflare 反向代理**的方式将请求转发到原有后端？
- **鉴权方式**：现有请求通过签名（`MailKey.signData`）进行鉴权，迁移后鉴权机制是否需要调整？
- **客户端改动范围**：扩展侧（`background.ts` 中的 `BMRequestToSrv`、`getContactSrv`）需要哪些改动？是否需要更新服务地址配置方式？
- **兼容性过渡**：迁移期间如何保证老版本扩展用户不中断服务？

## 4. 非功能性需求

### 4.1 稳定性
- Bmail 核心功能（加密、解密、密钥管理）不依赖任何第三方网站的 DOM 结构
- 网站改版后，最多只影响适配器层的发送动作，不影响 Bmail 的核心内容管理

### 4.2 用户体验
- Bmail 自有面板的外观需与品牌保持一致，独立于各邮件网站的视觉风格
- 面板的触发、展示、关闭逻辑需流畅，不阻塞用户对原生邮件功能的使用
- 面板应该是非侵入式的，用户可以随时在 Bmail 面板和原生界面之间切换

### 4.3 兼容性
- 继续支持：Gmail、网易邮箱（163/126）、QQ 邮箱、Outlook Live
- 新架构设计需考虑后续扩展新网站时的接入成本，理想情况是只需实现一个标准适配器接口即可接入

### 4.4 安全性
- 加密/解密逻辑继续在 background service worker 中执行，不暴露在页面上下文
- 密钥不离开 background 层

## 5. 决策记录（原开放性问题）

### 问题一：邮件阅读与解密的界面方案 ✅ 已决策

**背景**：用户在邮件列表中点开一封 Bmail 加密邮件时，邮件正文是密文（JSON 格式的加密数据）。原方案是在原生阅读界面内注入解密按钮，解密后直接替换原生界面中的密文内容。

**决策结论：保留注入原生界面方案，进行语义化瘦化改造。**

**放弃独立 UI + 官方 API 的原因**：
- Gmail API / Outlook Graph API 的申请审批流程复杂（可能需要数周），且需通过 Google/Microsoft 的数据安全合规审查
- QQ 邮箱、网易邮箱个人版无公开 Web API，只能通过 IMAP/SMTP，而浏览器扩展无法直接使用 TCP Socket，需要额外的中继服务
- 整体接入成本和周期无法满足当前阶段的交付目标

**放弃 DOM 扫描方案的原因**：
- Gmail 使用虚拟滚动列表，无法全量扫描
- 邮件列表页通常只展示摘要，Bmail 密文（JSON 格式）在摘要中可能被截断，无法可靠识别

**选择保留注入的理由**：
- 阅读侧注入比撰写侧简单得多：只需定位邮件正文容器，在其前插入解密按钮（1-2 个选择器即可）
- 语义化改造后（使用 `[role]`、`[aria-label]`、`[data-testid]` 替代样式类名），维护成本显著降低
- FAB 备用入口作为兜底，注入失败不影响用户使用

**落地要求**：
- 每个平台只保留 1-2 个稳定的语义化选择器来定位邮件正文
- 解密按钮注入逻辑统一提取到 `content_common.ts`，各平台只提供定位函数
- FAB 提供「解密当前邮件」备用入口

### 问题二：Bmail 自有撰写面板的挂载与展示策略

**背景**：Bmail 面板需要以某种形式展示在用户正在使用的邮件网站上。

**待决策点**：
- 面板是以 **Shadow DOM** 的方式隔离样式，还是以 **iframe** 的方式完全隔离（指向 extension 内部页面）？
- 面板的展示形态：**侧边抽屉**（不遮挡原生界面）、**模态弹窗**（全屏覆盖）、还是**内嵌 Panel**（嵌入到原生页面某区域）？
- 如何处理面板与 background service worker 的消息通信（特别是 iframe 方案下的跨域消息）？

### 问题三：适配器触发原生发送的可靠性

**背景**：新架构中"触发原生发送动作"是适配器最关键的接口。

**待决策点**：
- 如果网站原生发送按钮无法通过 `.click()` 触发（某些网站有防护），是否有备用方案？
- 对于 QQ 邮箱老版本这类使用 iframe 作为编辑区的场景，填充内容和触发发送的策略是否与新版不同？

### 问题四：Cloudflare 服务迁移方案

**待决策点**（见 3.5.3 节）：
- Workers 部署 vs 反向代理，哪种方案更适合当前服务特性？
- 客户端改动范围和迁移节奏如何规划？

## 6. 当前架构与新架构的对比

| 维度 | 当前架构（V1） | 新架构（V2） |
|-----|-------------|------------|
| **邮件编写界面** | 在原生编辑器内注入按钮和逻辑 | Bmail 自有撰写面板 |
| **加密时机** | 点击"加密发送"时，在原生编辑区操作 | 在 Bmail 面板内完成加密，再填充 |
| **附件管理** | 拦截原生附件上传流程 | Bmail 面板内独立管理 |
| **发送动作** | 加密后直接模拟点击原生发送按钮 | 适配器负责填充+触发发送 |
| **邮件阅读** | 注入解密按钮到原生阅读区 | **保留注入，语义化选择器改造**（维护成本可接受） |
| **联系人服务** | 自有服务器直连 | 迁移至 Cloudflare |
| **第三方 DOM 依赖** | 大量硬编码 CSS 选择器 | 仅适配器层，最小化 |
| **多网站适配成本** | 每个网站独立一套完整逻辑 | 每个网站只需实现标准适配器接口 |
| **抗改版能力** | 弱（网站改版即失效） | 强（仅适配器层受影响） |

## 9. 附录：当前代码关键结构说明

供架构师参考，快速理解现有代码：

### 9.1 文件职责说明

| 文件 | 职责 |
|-----|-----|
| `content.ts` | 公共 Content Script，处理来自 inject.js 的消息，路由加解密请求到 background |
| `content_common.ts` | 各网站 Content Script 的公共工具函数（DOM 操作、加解密调用、联系人查询等） |
| `content_google.ts` | Gmail 专属适配逻辑 |
| `content_netease.ts` | 网易邮箱专属适配逻辑 |
| `content_outlook.ts` | Outlook 专属适配逻辑 |
| `content_qq.ts` | QQ 邮箱专属适配逻辑（含新旧两个版本） |
| `content_attachment.ts` | 附件加解密逻辑 |
| `background.ts` | Service Worker，处理加密、解密、签名、联系人查询、下载监听等核心逻辑 |
| `consts.ts` | 常量定义（MsgType 消息类型枚举等） |
| `setting.ts` | 配置管理，包含 `getContactSrv()` 获取服务器地址 |

### 9.2 关键消息类型（MsgType）

```typescript
MsgType.EncryptData           // 加密邮件正文
MsgType.DecryptData           // 解密邮件正文
MsgType.EmailAddrToBmailAddr  // 邮箱地址 → 区块链地址（联系人查询）
MsgType.CheckIfLogin          // 检查钱包是否已解锁
MsgType.IfBindThisEmail       // 检查当前邮箱是否已绑定
MsgType.BMailInbox            // 打开 Bmail 插件弹窗
MsgType.KeyForLocalApp        // 发送附件密钥给本地 App
```

### 9.3 核心加密流程（现有）

```
Content Script 获取收件人邮箱地址列表
       ↓
发送 EmailAddrToBmailAddr 到 background，查询区块链地址
       ↓
发送 EncryptData 到 background（携带区块链地址列表 + HTML 正文）
       ↓
background 调用 encodeMail() 完成加密，返回 JSON 密文
       ↓
Content Script 将密文写入原生编辑区，触发原生发送按钮
```

这一流程在 V2 中需要保留，仅调整"将密文写入原生编辑区"和"触发发送"的部分下沉到适配器层。

### 9.4 现有联系人服务调用链（Cloudflare 迁移参考）

```
background.ts
  └── searchAccountByEmails()
        └── BMRequestToSrv(API_Query_By_EMails, ...)
              └── getContactSrv()  ← 当前返回自有服务器地址（来自 setting.ts）
                    └── fetch(hostUrl + apiPath, ...)
```

迁移后，`getContactSrv()` 返回的地址将指向 Cloudflare 的 Workers 或代理域名，其余调用链保持不变（最小化客户端改动）。

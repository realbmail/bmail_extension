## 1. 整体规划概览

本次迭代旨在彻底剥离对第三方网站 DOM 的重度依赖，转向“自有面板 + 瘦适配器”架构。整体工期预估为 **6 周**，分为四个关键冲刺（Sprint）。

### 1.1 里程碑（Milestones）
*   **M1 (Week 1)：基础设施就绪。** 完成适配器标准接口定义、全局 FAB 挂载器与 Iframe 容器注入方案，并打通扩展端到宿主端的跨域通信。
*   **M2 (Week 2-3)：撰写面板 MVP。** 独立于任何第三方网站，完成 Bmail Iframe 内的富文本撰写、收件人解析、加密执行。
*   **M3 (Week 4-5)：全平台适配器改造。** 完成 Gmail、QQ、网易、Outlook 四大平台的瘦适配器对接及阅读侧语义化改造。
*   **M4 (Week 6)：云端演进与全功能联调。** 服务端 API 网关（Cloudflare）切换，端到端完整链路联调与性能保障。

### 1.2 任务依赖拓扑
```text
[T01 统一接口契约] ┬─► [T02 全局挂载器] ────────┬─► [T04 多平台适配器开发]
                   │                          │
                   └─► [T03 独立撰写面板开发] ─┘
                                              │
[T05 Cloudflare网关] ─► [T06 客户端服务切换]  ─┴─► [T07 端到端联调与回归]
```

## 2. 详细任务卡与验收卡拆解

以下任务拆解严格遵循架构师任务规范，开发人员需严格按 Task Card 进行开发，按 Acceptance Card 提交代码合并请求（PR）。

### 【Sprint 1】基础设施与契约定义

#### Task Card 01：定义瘦适配器（Thin Adapter）统一契约
*   **任务描述**：为所有第三方邮件网站抽象标准化的操作边界规范。设计一套基于 TypeScript 的 `Adapter` 接口，隔离不同邮箱站点的底层逻辑。
*   **技术方案**：定义包含 `提取写信环境()`、`填充密文并触发发送()`、`定位含密文的正文区()`、`运行环境监测()` 等关键抽象方法的 Interface。设计基于策略模式的发送状态回调协议。
*   **涉及文件**：新增 `src/adapters/AdapterInterface.ts` 及类型定义文件。
*   **依赖任务**：无。
*   **预估复杂度**：简单。
*   **注意事项**：接口中关于 `填充并发送` 的返回值设计，需包裹“成功/降级至手工触发/彻底失败”三种状态标识。

#### Acceptance Card 01：统一契约验收
*   **功能验收**：接口定义清晰，能涵盖目前四家厂商（包含 QQ 旧版 Iframe）的需求范围。
*   **代码质量**：TypeScript 泛型与入参/出参严格类型化，附带详尽的 JSDoc 架构注释。

---

#### Task Card 02：开发全局控制台（FAB）与 Iframe 沙箱注入器
*   **任务描述**：取代 V1 中的原生 UI 侵入，开发一个通用的 DOM 注入类，负责在目标页面挂载 Bmail 专属的控制入口和抽屉容器。
*   **技术方案**：
    1. 使用 **Shadow DOM** 技术向原生 `document.body` 插入一个悬浮按钮（FAB），彻底隔离原生网站 CSS 洗脱。
    2. 创建一个绝对定位的侧边栏 `Iframe` 元素，`src` 指向扩展内的 `compose.html`。
    3. 封装 `Iframe` 与宿主 `Content Script`（适配器层）基于 `window.postMessage` 的安全双向通信总线。
*   **涉及文件**：新增 `src/fab_injector/` 相关逻辑文件。
*   **依赖任务**：T01。
*   **预估复杂度**：中等。
*   **注意事项**：务必校验 `event.origin` 以防范外部消息仿冒攻击，同时处理焦点拦截问题，点击外部应自动收起抽屉。

#### Acceptance Card 02：沙箱注入器验收
*   **功能验收**：在四大邮箱页面均能稳定出 FAB 按钮，点击后平滑侧滑出 Iframe 面板；宿主站点的重构/CSS变动不影响 FAB 及侧边栏样式。
*   **安全检查**：`postMessage` 监听器严谨判断了数据 Payload 的校验签名/Origin 来源。

### 【Sprint 2】自有面板与核心撰写闭环

#### Task Card 03：Bmail 自有撰写面板（Compose Panel）开发
*   **任务描述**：在扩展隔离环境（`chrome-extension://`）下，完成具有 Bmail 原生体验的富文本撰写、收件人状态录入与一键加密发往宿主的闭环流程。
*   **技术方案**：
    1. 构建 `compose.html` 及对应的独立 JS 运行环境。
    2. 实现去格式化的富文本编辑区、附件拖拽上传区及多标签形态的收件人组件。
    3. 接收用户“加密发送”指令后，直连 Background 层调用密码学核心运算（直接调 `chrome.runtime.sendMessage`）。
    4. 将加密后的 JSON 密文通过前置封装的 `postMessage` 通道推向宿主适配器。
*   **涉及文件**：新增 `src/compose/compose_panel.ts`、`html/compose.html`、`css/compose.css` 等。
*   **依赖任务**：T02。
*   **预估复杂度**：复杂。
*   **注意事项**：密码学运算需保持异步非阻塞状态，上传巨大附件时不应卡死当前面板 UI。需与 T02 联通进度回调。

#### Acceptance Card 03：撰写面板验收
*   **功能验收**：独立窗口能完整跑通“拉取后台状态->自动补全联系人->大附件/长文本秒级加密->推送成功”的全工作流。
*   **性能要求**：富文本框丝滑，切换输入无迟滞卡顿，面板开启/关闭动画 60fps。

### 【Sprint 3】全平台适配器改造

#### Task Card 04：实施各平台瘦适配器（Gmail / Netease / QQ / Outlook）
*   **任务描述**：大范围清除 V1 在四个 `content_*.ts` 中的所有前端拼装和加解密逻辑组件，全部重构为针对 `T01 接口契约` 的简单物理动作执者。
*   **技术方案**：

    2. **内容填充**：利用 `execCommand` 或剪贴板事件模拟，将面板推过来的密文塞入原生邮件富文本框。
    3. **阅读侧语义化重构**：将以前抓取 DOM 的重度 class (`.a3s.aiL` 等)，全量置换为寻找 `[role="main"]`、`[aria-label="Email details"]` 这种防改版的语义化或测试属性标签。
*   **涉及文件**：新增 `src/adapters/adapter_*.ts` 家族，完全废除原 `content_*.ts`（或将其作为纯净入口）。
*   **依赖任务**：T01（接口约定即唯一约束）。
*   **预估复杂度**：复杂。
*   **注意事项**：由于老版 QQ 邮箱仍使用过时的 Iframe 编辑器结构，QQ 适配器内部必须含有新老版路由判断机制。

#### Acceptance Card 04：全平台适配器验收
*   **功能验收**：在不手动触碰原生窗口任何按钮的情况下，在 Bmail 侧边栏点“发送”，自动驱使 4 大邮件产商将邮件正式投递出去并退回列表视图。
*   **代码质量**：旧版巨型面条代码被物理铲除；每个适配器仅依赖原生站点的少数不到个位数的语义化 DOM ID/属性组合。

### 【Sprint 4】云端整合与质量收官

#### Task Card 05 & 06：云服务网关迁移与整合
*   **任务描述**：为增加后端防 DDoS 并降低网络时延，运用 Cloudflare Worker 创建反向代理，转移原先 Background 直连后端的请求。
*   **技术方案**：
    1. （无前端编码，平台操作）编写一个短小精悍的 Cloudflare JS Worker 进行白名单 API 的透明反向代理（Proxy）。
    2. 将 `src/setting.ts` 中的 `__officialContactSrv` 指向全新的 Worker CDN 边缘节点。
    3. 全面自测包括：绑卡、解绑、地址簿查询。
*   **预估复杂度**：简单。
*   **注意事项**：签名鉴权必须依然完整贯穿到 origin 源站，确保签名验伪机制没报 CORS 阻断失败。

#### Task Card 07：全量端到端（E2E）集成测试修缮
*   **任务描述**：模拟极其恶劣但真实的使用环境（弱网情况、未解密时强制点发送、大体积多重加密附件、对方并未注册 Bmail 强制发件等。
*   *具体不赘述，但涉及整体链路的熔断回调是否准确投射到用户 UI 上。*

## 3. 工作原则申明与工程纪律
1. **不做多余假设**：在写适配器时，绝不要尝试解析原生网站的 DOM 内容流来提取上下文。原站的职责仅仅是搬运密文。
2. **渐进式切换策略**：本地工程在开发 V2 期间，可通过 webpack 不同的入口或 feature flag（若适用），保证 V1 代码可在开发分支并行参考比对直到 V2 集成通过。
3. **安全自检**：`adapter` 中但凡用到 `innerHTML` 的地方，必须严防被宿主环境恶意 XSS 利用，原则上只能信任由 Bmail 扩展域发出的强加密包数据流。

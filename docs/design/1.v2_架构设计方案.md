## 1. 架构升级概述

### 1.1 架构核心哲学
本次架构升级（V2）的核心目标是解决 V1 架构中**“深度依赖外部不确定性（第三方网站 DOM 结构）”**的脆弱性问题。
V2 架构采用**“依赖反转”**与**“边界隔离”**的设计哲学：
*   **边界隔离**：将 Bmail 的核心价值（撰写、加解密、密钥管理）与第三方网站完全隔离，构建闭环的自有交互界面。
*   **依赖下沉与最小化**：将对第三方网站的依赖，从“全生命周期的 DOM 劫持”，降级并收敛为单一的“发送通道适配（瘦适配器）”。

### 1.2 系统上下文
Bmail 系统在 V2 架构下面临四个核心上下文环境：
1.  **用户**：通过扩展程序与 Bmail 的 UI 界面（撰写面板、悬浮按钮）交互。
2.  **宿主环境（第三方邮件网站）**：提供基础的网页运行容器，以及最终的邮件投递和邮件展示功能。
3.  **扩展后台（Background）**：系统的心脏，负责高安全级别计算与状态管理。
4.  **云端服务（Contact Service）**：提供区块链地址查询及账户维度的支持引擎。

## 3. 关键模块与技术决策 (ADR)

为支撑《V2_架构升级需求说明书》的目标，架构上做出如下关键技术决策和模块划分：

### 3.1 独立撰写面板的形态与挂载模型
*   **决策**：采用 **Web Extension Iframe** 作为撰写面板载体，采用 **Shadow DOM** 作为入口悬浮按钮（FAB）的载体。
*   **架构依据**：
    *   **Iframe 方案**可以使得撰写面板直接运行在 `chrome-extension://` 协议下。这带来了两点极大优势：一是**样式与 JS 运行环境的绝对隔离**，彻底杜绝宿主页面的 CSS 污染与 JS 冲突；二是**具备完整的 Chrome API 权限**，无需通过耗时的 Content Script 进行消息中转即可直连 Background。
    *   **Shadow DOM 方案**适合轻量的挂载点（FAB 按钮），直接挂载于 `document.body`，大幅减少 DOM 查找匹配不稳定带来的挂载失败。
*   **展示形式**：以侧边抽屉形式从屏幕边缘滑出，满足“非阻塞性”阅读与引用参照需求。

### 3.2 瘦适配器（Thin Adapter）设计规范
*   **决策**：提取并固化第三方网站的交互行为边界，定义标准的抽象接口（Interface）。
*   **架构依据**：将对易变系统的依赖控制在最小范围。适配器**只负责干两件事**：
    1.  **数据的搬运工**：负责将 Iframe 传出的密文塞进原生的富文本编辑区。
    2.  **动作的触发器**：负责利用各种手段（DOM 点击、事件合成、按键模拟，乃至退化人工点击提示）触发网站的原生发送流程。
*   **健壮性要求**：适配器必须实现**多级降级策略**。当最高优先级的 DOM 点击失效时，必须无缝退化到事件驱动或引导用户手工点击完成最后一步，确保流程断点不丢失用户数据。

### 3.3 阅读侧的轻量注入与语义化改造
*   **决策**：维持在内容区注入解密按钮的模式，但全面弃用传统的基于层级与复杂 CSS 组合的选择器。
*   **架构依据**：考虑到完全替代阅读器 API 接入成本过高且不现实，保留 DOM 注入是唯一可行解路线。其策略重点转为对**语义化标签（如 Role、Aria-label、Data-testid 等）**的精准识别验证。由于第三方服务为兼顾无障碍访问（A11y）或自身自动化测试考量，这类属性的更迭频率远低于视觉样式（Class），借此大幅提高系统寿命。

### 3.4 后端服务的云端化演进 (Cloudflare 迁移)
*   **决策**：采用 **Cloudflare Workers 作为反向代理（API 网关）**。
*   **架构依据**：
    *   **最小改动原则**：系统当前的主要痛点为“接入层可用性”与“访问速度优化”。在此阶段重写已有后端业务逻辑至 Workers 存在极大的业务一致性及数据迁移风险。
    *   借由 API 网关模式，可在**零业务逻辑重构、零鉴权规则修改**的情况下，剥离域名解析与源站暴露的风险，利用边缘网络大幅优化全球客户端连通率。
    *   为未来业务逐步迁移至 Serverless 架构（若需）预留分流路由能力。

## 4. 核心工作流设计

### 4.1 邮件撰写与发送状态机

```text
[启动态] 点击悬浮FAB -> 展开扩展 Iframe (Bmail 独立环境)
  ↓
[草稿态] 用户在独立环境中录入收件人、标题、正文文本
  ↓
[校验态] 拦截并预检：
         |- Background.CheckAuth: 钱包是否解密登录？
         |- Background.QueryContact: 收件人公钥聚合检索
  ↓
[加密态] 扩展 Iframe -> 内部直通 -> Background (执行加密任务) 
  ↓
[移交态] 扩展 Iframe -> PostMessage 跨域通信 -> Content Script (宿主页面)
         宿主适配器接收密文，执行原生填充逻辑与事件触发布局
  ↓
[终结态] 发送结果上报，执行回收操作 (清空上下文，折叠 Iframe 面板)
```

### 4.2 API 网关代理流程

对于任何发往原来联系人系统的请求，系统交互统一收口：
`Iframe / ContentScript` -> `Background.BMRequestToSrv` -> `读取配置 (指向 Cloudflare 域名)` -> `CF_Worker 边缘拦截并转发` -> `Origin Backend 校验并响应`。

## 5. 架构边界与非功能性保证

1.  **安全性（Security）**
    *   数据隔离：所有用户私钥与核心加密算法固化于 Background Worker，扩展内部通信不包含敏感私钥流出；
    *   防篡改与劫持：Iframe 的跨域执行确保宿主网站（即使被 XSS 等方式渗透）也无法突破沙箱读取 Bmail 的撰写内容明文。
2.  **可用性（Availability）**
    *   适配器健壮设计要求在网站严重改组时启动“兜底提示”，提示用户采取最后一步人工兜底；
    *   阅读侧解密挂载点如若找不到，使用 FAB 上的全局“解密当前页面”进行系统级灾备入口接入。
3.  **可维护性（Maintainability）**
    *   废弃海量的 `content_google.ts` / `content_netease.ts` 等全生命周期的巨型文件；
    *   所有的网站支持仅体现为一个继承于 `AdapterInterface` 的简易类，新增一家邮件支持厂商的研发周期理论上将被压缩至 1 天之内。

# Sprint 2: 写信面板开发

**所属规划阶段**: 里程碑 M2 (写信面板开发)
**文档状态**: 📋 待启动
**依赖**: Sprint 1 已完成

本阶段将在 Sprint 1 建立的基础设施之上,开发完整的写信面板功能。重点是在 Iframe 沙箱内实现富文本编辑、加密逻辑和收件人管理,完成从用户输入到加密邮件发送的完整流程。

---

## 一、整体节点目标

1. **富文本编辑器集成**: 在 compose.html 中集成成熟的富文本编辑器,支持基本的文本格式化
2. **加密逻辑实现**: 实现邮件内容的加密功能,生成加密包
3. **收件人管理**: 实现收件人输入、验证和管理功能
4. **完整发送流程**: 打通从用户输入到通过 postMessage 传递加密包给适配器的完整链路

---

## 二、任务排期与拆解

### 任务 2.1: 富文本编辑器集成

**主要执行动作**:
1. 选择并集成轻量级富文本编辑器(推荐 Quill 或 TinyMCE)
2. 在 compose.html 中实现编辑器 UI
3. 实现基本的文本格式化功能(粗体、斜体、列表等)
4. 实现编辑器内容的获取和设置

**预期输出文件**:
- `* extension/html/compose.html` - 添加编辑器 UI
- `* src/compose.ts` - 集成编辑器逻辑
- `+ src/compose/editor.ts` - 编辑器封装模块
- `* extension/manifest.json` - 添加编辑器库资源

**验收标准**:
- 编辑器能正常显示和输入
- 支持基本的文本格式化
- 能获取 HTML 格式的编辑器内容

---

### 任务 2.2: 收件人管理功能

**主要执行动作**:
1. 实现收件人输入框 UI
2. 实现邮箱地址验证逻辑
3. 支持多个收件人的添加和删除
4. 实现收件人列表的展示

**预期输出文件**:
- `+ src/compose/recipient_manager.ts` - 收件人管理模块
- `* extension/html/compose.html` - 添加收件人 UI
- `* src/compose.ts` - 集成收件人管理

**验收标准**:
- 能输入和验证邮箱地址
- 支持添加多个收件人
- 能删除已添加的收件人
- 显示收件人列表

---

### 任务 2.3: 加密逻辑实现

**主要执行动作**:
1. 集成加密库(使用现有的加密方案)
2. 实现邮件内容加密功能
3. 生成符合 BMail 格式的加密包
4. 实现加密状态的 UI 反馈

**预期输出文件**:
- `+ src/compose/encryptor.ts` - 加密逻辑模块
- `* src/compose.ts` - 集成加密功能
- `* extension/html/compose.html` - 添加加密状态 UI

**验收标准**:
- 能正确加密邮件内容
- 生成的加密包格式正确
- 加密过程有 UI 反馈
- 加密失败有错误提示

---

### 任务 2.4: 发送流程集成

**主要执行动作**:
1. 实现发送按钮和发送逻辑
2. 整合收件人、主题、内容和加密
3. 通过 postMessage 将加密包发送给宿主页面
4. 实现发送状态的 UI 反馈

**预期输出文件**:
- `+ src/compose/sender.ts` - 发送流程模块
- `* src/compose.ts` - 集成发送流程
- `* src/fab_injector/message_bus.ts` - 处理 SEND_EMAIL 消息
- `* extension/html/compose.html` - 完善发送 UI

**验收标准**:
- 点击发送按钮能触发完整流程
- 加密包通过 postMessage 正确传递
- 宿主页面能接收到加密包
- 发送过程有完整的 UI 反馈

## 四、技术方案建议

### 富文本编辑器选择
推荐使用 **Quill**:
- 轻量级,易于集成
- API 简洁,文档完善
- 支持自定义工具栏
- 可以通过 CDN 引入,无需构建步骤

### 加密方案
继承现有 BMail 的加密逻辑:
- 使用现有的加密库和算法
- 保持加密包格式的兼容性
- 确保加密强度符合安全标准

### 消息通信协议
扩展 Sprint 1 定义的消息类型:
```typescript
interface SendEmailMessage {
  type: 'SEND_EMAIL';
  payload: {
    recipients: string[];
    subject: string;
    encryptedContent: string;
    metadata?: Record<string, any>;
  };
}
```

## 六、风险识别

### 技术风险
1. **富文本编辑器兼容性**: 不同编辑器在 Iframe 中的表现可能不同
   - 缓解措施: 优先选择经过验证的成熟方案

2. **加密性能**: 大量文本加密可能影响性能
   - 缓解措施: 实现加密进度反馈,考虑 Web Worker

3. **跨域通信限制**: postMessage 传递大数据包可能有限制
   - 缓解措施: 测试大邮件场景,必要时分块传输

### 进度风险
1. **加密逻辑复杂度**: 如果现有加密方案需要大幅调整
   - 缓解措施: 提前评估现有代码,必要时简化需求

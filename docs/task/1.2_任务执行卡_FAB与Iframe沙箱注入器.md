## 任务目标

开发基于 Shadow DOM 的悬浮控制按钮（FAB）和 Iframe 沙箱容器，建立安全的跨域通信机制，为后续的写信面板提供挂载基础。

---

## 任务描述

1. 使用 Shadow DOM 技术创建一个不受宿主页面样式污染的悬浮按钮（FAB）
2. 构建侧边栏 Iframe 容器，用于加载扩展内部的写信面板
3. 建立安全的 postMessage 通信总线，实现 Iframe 与宿主页面的双向通信

---

## 技术方案

### 1. FAB 悬浮按钮（Shadow DOM）

#### 实现要点
- 使用原生 JavaScript 向 `document.body` 动态插入 Shadow DOM 节点
- 按钮位置：屏幕右下角固定定位（类似悬浮加号 ➕）
- 样式隔离：利用 Shadow Root 确保原生站点的 CSS 无法污染按钮样式
- 交互：点击按钮时唤起 Iframe 侧边栏

#### 技术细节
- 使用 `attachShadow({ mode: 'closed' })` 创建封闭的 Shadow DOM
- 按钮样式使用内联或独立的 CSS 文件
- 确保按钮在页面滚动时保持固定位置
- 处理不同屏幕尺寸的响应式布局

### 2. Iframe 侧边栏容器

#### 实现要点
- Iframe 的 `src` 指向 `chrome-extension://<id>/html/compose.html`
- 初始状态：隐藏在视觉区域外（使用 CSS transform）
- 唤起动画：使用 CSS translate 实现侧滑效果
- 尺寸：占据屏幕右侧一定宽度（如 400px），高度 100vh

#### 技术细节
- Iframe 需要设置合适的 sandbox 属性（如果需要）
- 使用 CSS transition 实现平滑的滑入/滑出动画
- 考虑移动端和小屏幕的适配
- 提供关闭按钮或点击遮罩层关闭的交互

### 3. 安全通信总线（Message Bus）

#### 宿主侧（fab_injector.ts）
- 监听 `window.addEventListener('message', ...)` 接收 Iframe 消息
- **安全校验**: 严格检查 `event.origin`，仅响应来自 `chrome-extension://` 的消息
- 消息处理：接收加密完成的数据包，调用当前激活的 `MailSiteAdapter.fillAndSend()`
- 错误处理：对非法来源的消息进行拒绝和日志记录

#### Iframe 侧（compose.html）
- 使用 `window.parent.postMessage()` 向宿主发送消息
- 消息格式：定义统一的消息协议（如 `{ type: 'SEND_EMAIL', payload: {...} }`）
- 此阶段仅需验证通信连通性，具体业务逻辑在 Sprint 2 实现

#### 消息协议设计
```typescript
interface Message {
  type: 'SEND_EMAIL' | 'CLOSE_PANEL' | 'READY' | ...;
  payload: any;
}
```

### 4. Manifest 配置

#### 必须修改 manifest.json
- 将 `compose.html` 添加到 `web_accessible_resources` 中
- 示例配置：
```json
{
  "web_accessible_resources": [{
    "resources": ["html/compose.html"],
    "matches": ["<all_urls>"]
  }]
}
```
- 如果不配置，浏览器的 CSP 会直接拦截 Iframe 加载

## 注意事项

- **样式隔离**: 必须使用 Shadow DOM，确保 FAB 不受宿主页面样式影响
- **安全第一**: `event.origin` 校验是安全的关键，必须严格实现
- **性能考虑**: Iframe 的创建和销毁要考虑性能影响，避免内存泄漏
- **兼容性**: 测试不同浏览器（Chrome, Edge）和不同邮箱站点的兼容性
- **用户体验**: 动画要流畅，按钮位置不能遮挡重要内容
- **CSP 限制**: 确保 manifest.json 配置正确，否则 Iframe 无法加载
- **错误处理**: 对通信失败、Iframe 加载失败等异常情况要有完善的错误处理

---

## 技术风险

1. **Shadow DOM 兼容性**: 虽然现代浏览器都支持，但要注意边界情况
2. **CSP 策略冲突**: 某些邮箱站点可能有严格的 CSP，需要测试验证
3. **消息丢失**: postMessage 在某些情况下可能丢失，需要考虑重试机制
4. **Z-index 冲突**: FAB 的层级要足够高，避免被宿主页面元素遮挡

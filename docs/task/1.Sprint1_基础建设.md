# 当前阶段 (Sprint 1) 工作目标：基础设施与契约定义

**所属规划阶段**：里程碑 M1 (M1: 基础设施就绪)
**文档状态**：✅ 执行中

本阶段是 V2 架构演进的基石，重点在于完成**系统边界的确立**和**扩展UI挂载模式的变更**。该阶段主要涵盖 `2.v2_开发规划.md` 中的 **Task Card 01**（瘦适配器统一契约） 和 **Task Card 02**（全局控制台 FAB 与 Iframe 沙箱注入）。不涉及具体厂商业务逻辑的实现以及写信面板界面的开发。

---

## 一、整体节点目标

1.  **契约统一化**：出具 V2 阶段核心的 `Adapter` TypeScript 接口文件，终结过去针对每个站点单独写无序逻辑的历史。
2.  **挂载沙箱化**：不再将按钮通过复杂的类名查询注入到宿主的 DOM 层级中，而是利用 Shadow DOM 和 Iframe 在页面顶层构建自己的隔离容器。
3.  **打通跨域通信流**：保证 Bmail Iframe 和宿主 Adapter 之间能通过安全的双向 `postMessage` 信道通信。

---

## 二、任务排期与拆解 (Task Breakdown)

### 任务 1：统一瘦适配器规范（Adapter Interface）
**对齐规划**：Task Card 01
**主要执行动作**：
1.  在 `src/adapters/` 目录下（如无该目录则新建），定义核心契约文件 `AdapterInterface.ts`。
2.  声明一个抽象接口 `MailSiteAdapter`。要求所有的下游厂商（Gmail, Outlook, QQ 等）都必须 implement 这个接口。

**接口应当包含的核心要素约束**：
*   `readCurrentEnvironment()`：获取当前邮箱的地址和运行时状态。
*   `fillAndSend(ciphertext: string, subject: string): Promise<SendResult>`：将从 iframe 传来的加密包填充给原生编辑器，并尝试多重降级触发“发送”操作。
*   `findEncryptedContainers(): HTMLElement[]`：用于阅读侧，定位到当前被 Bmail 加密过的只读正文区，以便注入“解密”按钮。

**预期输出文件**：
*   `+ src/adapters/AdapterInterface.ts`

---

### 任务 2：开发 FAB（悬浮控制按钮）与 Iframe 沙箱注入器
**对齐规划**：Task Card 02
**主要执行动作**：
1.  **Shadow DOM 撰写 FAB 触发器**：利用原生 JS 编写一套向 `document.body` 中动态插入一个基于 Shadow DOM 的控制按钮逻辑（类似屏幕右下角的悬浮加号➕），因为使用了 Shadow Root，原生站点的 CSS 无法污染这个按钮。
2.  **构建侧边栏 Iframe 容器**：点击 FAB 时，唤起一个隐藏在视觉外的 Iframe（`src` 为 `chrome-extension://<id>/html/compose.html`）。配合 CSS translate 实现一种屏幕侧滑出的视效。
3.  **建设安全的通信总线 (Message Bus)**：
    *   在宿主的 `fab_injector.ts` 中，监听 `window.addEventListener('message', ...)` 准备接收 Iframe 抛出的加密完成包裹。
    *   在内部，对 `event.origin` 进行强校验，确保仅响应来自自身 chrome-extension 环境的数据推送。然后将数据包裹派发给当前激活的 `MailSiteAdapter.fillAndSend()`。

**预期输出文件**：
*   `+ src/fab_injector/fab_injector.ts` (主挂载与通信逻辑)
*   `+ src/fab_injector/fab.css` (独立的内部样式，若适用)
*   `+ html/compose.html` (此阶段仅需建出空壳 HTML 验证 iframe 的白屏加载即可，后续 Sprint 2 填补富文本面板组件)
*   `* manifest.json` (必须修改以将 compose.html 暴露在 `web_accessible_resources` 中，否则跨域 Iframe 将被浏览器的 CSP 直接拦截)

## 三、开发自测项 (验收基准 AC)

在向主干合并前，负责人应比对下列表格进行人工核验：

| 验证模块 | 校验步骤 | 期望结果 |
| :--- | :--- | :--- |
| **T01 类型安全** | 执行 TypeScript 编译自检 (`tsc --noEmit`) | `AdapterInterface` 泛型、联合类型及 Promise 抛出定义均无报错。 |
| **T02 挂载兼容性** | 打开 Gmail, QQ邮箱, 网易163页面 | 浏览器右下角均能一致、不受页面滚动影响地出现 FAB 入口按钮。 |
| **T02 布局隔离** | 打开浏览器开发者工具 (DevTools)，在控制台强行改页面字体大小或加上全局 `!important` 污染颜色。 | FAB 悬浮按钮由于被封装在 Shadow Root 中，完全不随原生页面崩溃。 |
| **T02 通信连通性** | (Mock) 从 iframe 控制台手动 `postMessage` 一个“触发投递假数据”。 | `fab_injector` 环境能准确监听到该数据并无跨域抛错。 |
| **T02 安全基准** | (Mock) 通过另一个非扩展域名的控制台随意 `postMessage` 数据。 | `fab_injector` 识别 `event.origin` 非法后予以拒绝（抛弃）。 |
